# Launch Readiness Score

NutriTail should not treat launch progress as a feeling or as a count of merged
pull requests. The project readiness score is evidence-based and comes from the
current QA gates, live route checks, chatbot recommendation audits, and
post-deploy reports.

## Current Target

- 95/100 is the beta-launch target.
- 90/100 means the product is broadly usable, but still needs focused live QA.
- Below 90/100 means launch work should stay on reliability, chatbot accuracy,
  data quality, or customer flow before new features.

## What Counts Toward The Score

The authoritative rollup is generated by:

```bash
npm.cmd run qa:live-readiness-dashboard
```

The dashboard writes:

```text
reports/live_readiness_dashboard.md
```

That report includes:

- 95/100 readiness score
- minimum readiness score
- customer-ready core status
- full OpenAI proof status
- core evidence score
- advisory evidence score
- stale report warnings
- next report to refresh

## Score Model

The live readiness score is intentionally split:

- 90% core evidence
- 10% advisory evidence

Core evidence is blocking. It covers:

- public launch routes
- Food V2 live routes
- account progress routes
- customer chatbot flow links
- production OpenAI environment presence
- OpenAI food brand guard
- chatbot live QA dashboard

Advisory evidence is confidence-building. It covers:

- OpenAI intake smoke
- authenticated account chatbot extract route

Skipped advisory checks do not fully block local development because they need
local-only credentials, but they keep the score from being treated as final
OpenAI proof.

## Why The Percentage May Not Move

The score should only move when a launch risk is measurably reduced. Examples:

- A new PR adds customer copy but no QA coverage: confidence improves, score may
  not move.
- A PR makes chatbot answers prettier but does not affect recommendation
  correctness: UX improves, score may not move.
- A PR adds a new guard to `qa:ci-readiness`: score confidence improves because
  future regressions are harder.
- A post-deploy live check is stale: score can stay flat or drop until the
  report is refreshed.

## What Moves 94 Toward 95

The last launch points should come from stronger proof, not more polish:

- Run and refresh dog/cat live QA after recommendation changes.
- Run OpenAI intake smoke with a local secret file.
- Run authenticated chatbot extract live route with a local account cookie file.
- Keep customer-facing recommendations free of internal scores, source tiers,
  review status, and missing-field internals.
- Keep Food V2 ranking guards passing for allergy, size, life stage, urinary,
  renal, senior, puppy/kitten, weight, and preference cases.
- Verify post-deploy freshness after every production deploy touching chatbot,
  Food V2, account, report, or public launch pages.

## Practical Progress Bands

Use these bands when explaining project status:

| Band | Meaning | Main Work |
| --- | --- | --- |
| 70-80% | Foundations are present, but UX/data/QA are still uneven. | Build missing flows and data architecture. |
| 80-90% | Product works, but launch confidence is not proven enough. | Fix recommendation errors, reports, account flow, and admin QA. |
| 90-94% | Beta-ready shape, with remaining proof gaps. | Live QA, OpenAI proof, customer polish, post-deploy evidence. |
| 95%+ | Beta launch candidate. | Monitor, collect feedback, fix real user issues quickly. |

## Human Summary

When the user asks "what percentage are we at?", answer from current evidence:

1. Check `reports/live_readiness_dashboard.md` if it is fresh.
2. Mention whether the score is current or stale.
3. Explain the biggest blocker to the next point.
4. Avoid raising the percentage just because a PR merged.
